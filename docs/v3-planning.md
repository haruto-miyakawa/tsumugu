# つむぐ v3.0 仕様書：READMEから自動入力機能

このドキュメントは、つむぐ v3.0 の最優先機能「READMEから自動入力」の仕様書である。Claude Code に投げて実装を進めるための完全な指示書として機能する。

## 1. このフェーズで作るもの

`/generate` ページに「READMEから自動入力」機能を追加する。ユーザーが README（または任意の Markdown / テキスト）と「プラットフォーム」（note / Qiita / Zenn）を入力すると、AI が解析して既存の生成フォームの各フィールドを自動入力する。ユーザーは内容を確認・修正してから、通常通り記事生成を実行する。

既存の生成フローを一切壊さず、入力体験だけを劇的に改善する設計である。

## 2. v2.0 までの完成状況（前提条件）

このフェーズに着手する時点で、以下は完成している:

- v1：Phase 1〜4（スタイル設定、スタイルAI解析、記事生成、整形・プレビュー）
- v2.0：デザイン刷新完了（デザイントークン、マスコット、UIプリミティブ、全画面のトーン統一）

つむぐは Next.js 16 + React 19 + TypeScript + Tailwind CSS v4 + Tiptap 3 + Anthropic Claude API で動作している。

## 3. ユーザーフロー

ひとつ、ユーザーが `/generate` を開く。
ふたつ、既存フォーム（テーマ、エピソード、ジャンル、口調、構成、長さ、ターゲット読者、追加メモ）の上に「READMEから自動入力」というボタンが表示されている。
みっつ、ボタンを押すと、モーダルが開く。モーダルには3つの要素がある:
  - プラットフォーム選択（note記事 / Qiita記事 / Zenn記事、ラジオボタン）
  - README入力欄（テキスト貼り付け、または .md / .txt ファイルをドラッグ&ドロップ）
  - 「解析する」ボタン
よっつ、「解析する」を押すと、つむぎマスコットが thinking 顔に変わり、ローディング表示が出る。
いつつ、解析が完了すると、モーダルが閉じて、既存フォームの各フィールドに値が自動入力される。
むっつ、ユーザーは内容を確認・修正できる。気に入らないフィールドだけ手で直す、という体験。
ななつ、いつも通り「生成する」ボタンを押すと、既存の `/api/generate` で記事生成が走る。

## 4. UIの設計

### モーダル全体

- 背景：半透明オーバーレイ（`bg-ink/50` 程度）
- モーダル本体：白背景、`shadow-pop`、`rounded-sm`、最大幅 640px、中央配置
- 上部にタイトル「READMEから自動入力」（Noto Serif、20px）
- 右上に閉じる × ボタン（`Icon name="close"`、ghost）

### プラットフォーム選択

```
プラットフォーム
[●] note記事   [○] Qiita記事   [○] Zenn記事
```

- ラベル：sans 12px、`text-mute`
- 各オプションは `Chip` コンポーネント、`active` 状態で accent ピンク背景

### README入力欄

```
README または任意のMarkdown
┌─────────────────────────────────────┐
│ ここに貼り付け、または               │
│ .md / .txt ファイルをドラッグ        │
│                                     │
│ （textarea）                        │
└─────────────────────────────────────┘
3,420 字
```

- ラベル：sans 12px、`text-mute`
- textarea：最低 8行、最大 16行、可変リサイズ
- ボーダー：`border-rule`、focus 時 accent
- ドラッグ&ドロップ対応：ファイル投下で内容を textarea に読み込む
- 下部に文字数カウンタ（mono、mute）

### 「解析する」ボタン

- `Btn kind="primary" size="lg"`、右に矢印アイコン
- 解析中は `disabled`、テキスト「解析中…」、つむぎマスコット 32px が thinking 顔で隣に表示

### 解析後のフィードバック

- モーダル閉じる
- 既存フォームの各フィールドに値が入る（フェードイン演出）
- ページ上部にトースト風通知「READMEから自動入力しました。内容を確認・編集できます」（つむぎ happy アイコン付き、3秒で消える）

## 5. APIの設計

### 新規API: `POST /api/generate/from-readme`

**リクエスト**

```typescript
type FromReadmeRequest = {
  readmeText: string;      // README本文
  platform: 'note' | 'qiita' | 'zenn';
};
```

**レスポンス**

既存の `GenerateForm` で使われている入力型に合わせる。仮の型：

```typescript
type FromReadmeResponse = {
  theme: string;                    // テーマ（1行）
  episode: string;                  // エピソード・ネタ（自由記述）
  genre: 'エッセイ' | 'ハウツー' | '体験記' | 'レビュー' | 'コラム';
  tone: 'です・ます' | 'カジュアル' | '詩的';
  structure: '起承転結' | '結論先出し' | '散文';
  length: 'short' | 'normal' | 'long';
  targetReader: string;             // ターゲット読者（自由記述）
  additionalNotes: string;          // 追加メモ（自由記述）
};
```

各フィールドは、既存の `/generate` フォームと同じ enum 値を使う。実装時、`components/generate/GenerateForm.tsx` の型を参照して合わせること。

**内部処理**

ひとつ、文字数チェック：`readmeText.length` を測定し、後述の「5. APIの設計（補足）：文字数アラート」に従って、フロント側でアラートを出すか判断する。アラートはユーザーが「続行」を選んだ場合のみ、API へ進む。
ふたつ、長さに応じた処理：すべての長さで、Claude Haiku による要約を経由する設計とする（精度優先）。Claude Haiku に「このREADMEを2000字以内に要約してください。記事化のために以下を残す: コンセプト、技術選定の理由、苦労した点、解決した課題、想定読者」と指示。
みっつ、構造化抽出：Claude Sonnet 4.6 の `tool_use` を使って、要約結果から上記レスポンス型を完全に満たすJSONを生成する。プラットフォームごとにシステムプロンプトを切り替える。

> **注**：当初は「5000字未満は要約パスをスキップ」という分岐を検討したが、精度を重視して全件要約を経由する方針に変更した。コストはトレードオフだが、note / Qiita / Zenn の特徴を最大限に反映するため。

**補足：文字数アラート（3段階）**

長文READMEを投げると Claude API のコストが線形に増加する。商用利用を見越して、ユーザー（または未来の自分）に「これは長文なので時間とコストがかかります」と知らせる仕組みを入れる。

アラートはフロント側で実装する。`FromReadmeModal.tsx` の「解析する」ボタン押下時、文字数を測定して以下の3段階で警告:

| 文字数 | 警告レベル | UI挙動 |
|---|---|---|
| 0〜4,999字 | 警告なし | そのままAPIへ |
| 5,000〜9,999字 | 軽い注意 | アイコン + 短文：「長めのREADMEです。解析に少し時間がかかります（〜10秒）」※「続行」ボタンで進める。キャンセル可能 |
| 10,000〜14,999字 | 中程度の注意 | カード型の警告：「READMEがかなり長いです。コストが増えます（参考：通常の約2倍）。要点だけ抜粋して投げると精度も上がります」※「続行」「キャンセル」を選択 |
| 15,000字以上 | 強い警告 | カード型の強い警告 + つむぎ thinking 顔：「READMEが非常に長いです（◯◯字）。コストが増え、要約品質も落ちる可能性があります。可能であれば要点を抜粋することを推奨します」※「それでも続行」「キャンセル」を選択 |

設計の考え方:

ひとつ、**ユーザーに主導権**。警告を出すが、ブロックはしない。「続行」を選べば必ず処理する。
ふたつ、**段階的に強める**。5000字でいきなり強警告だとUXが悪い。10000、15000と段階を踏む。
みっつ、**つむぎは黒子**。最も強い警告のときだけ、つむぎマスコットを thinking 顔で添える。観察した事実を伝える口調。
よっつ、**コストの可視化**。15000字超でも金額を示さない。「通常の約2倍」のような相対表現で。具体的な金額表示は商用化フェーズで検討。

実装ヒント:

```typescript
// components/generate/FromReadmeModal.tsx 内
const getWarningLevel = (length: number): 'none' | 'light' | 'medium' | 'strong' => {
  if (length < 5000) return 'none';
  if (length < 10000) return 'light';
  if (length < 15000) return 'medium';
  return 'strong';
};

// 「解析する」ボタン押下時
const handleAnalyze = () => {
  const level = getWarningLevel(readmeText.length);
  if (level === 'none') {
    callApi();
  } else {
    setShowWarning(level); // 警告ダイアログを表示
  }
};
```

警告ダイアログは、モーダル内に重ねて表示する小さなカード。「続行」「キャンセル」の2ボタン構成。

**プラットフォーム別のシステムプロンプト**

note記事モード:
```
あなたはnote記事作成の編集者「つむぎ」です。
入力されたREADMEから、note向けの記事を書くための素材を抽出してください。
note記事の特徴:
- 物語性、開発の裏側、モチベーションを重視
- エッセイ調、読みやすい
- 読者は技術者だけでなく、開発に興味がある一般人も含む
出力する各フィールドは、note記事として最適な内容に整えてください。
特に:
- genre: 「エッセイ」「体験記」を優先
- tone: 「です・ます」「カジュアル」を優先
- structure: 「散文」「起承転結」を優先
```

Qiita記事モード:
```
あなたはQiita記事作成の編集者「つむぎ」です。
入力されたREADMEから、Qiita向けの記事を書くための素材を抽出してください。
Qiita記事の特徴:
- 実装の解説、コード例、つまずきポイントを重視
- 構造化された見出し、技術用語が頻出
- 読者は同業の開発者
出力する各フィールドは、Qiita記事として最適な内容に整えてください。
特に:
- genre: 「ハウツー」「レビュー」を優先
- tone: 「です・ます」を優先
- structure: 「結論先出し」を優先
```

Zenn記事モード:
```
あなたはZenn記事作成の編集者「つむぎ」です。
入力されたREADMEから、Zenn向けの記事を書くための素材を抽出してください。
Zenn記事の特徴:
- 技術的な深堀り、実装の工夫、設計判断を重視
- やや硬めの文体、エンジニアコミュニティ向け
- 読者は技術選定や設計に興味がある開発者
出力する各フィールドは、Zenn記事として最適な内容に整えてください。
特に:
- genre: 「ハウツー」「コラム」を優先
- tone: 「です・ます」を優先
- structure: 「結論先出し」「起承転結」を優先
```

## 9. 設計の哲学

このフェーズで守るべき哲学:

ひとつ、**既存フローを壊さない**。`/api/generate` には触らない。データモデルにも触らない。新機能は「既存フォームを楽に埋めるための補助ツール」として位置づける。

ふたつ、**ユーザーが主導権を持つ**。AI が解析した結果は「下書き」であり、ユーザーが確認・修正できる。AIが押し付けるのではなく、ユーザーの判断を補助する。

みっつ、**プラットフォームの個性を尊重**。note / Qiita / Zenn はそれぞれ文化が違う。同じREADMEから異なる素材が出るのが正解。

よっつ、**つむぎは黒子**。マスコットの存在感は維持しつつ、AIが「魔法」のように振る舞わない。観察した事実だけを伝える口調。
